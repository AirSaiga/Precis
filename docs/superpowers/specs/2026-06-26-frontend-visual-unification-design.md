# 前端视觉统一与面板修缮设计

> **日期**: 2026-06-26
> **状态**: 待评审
> **阶段**: Alpha
> **关联**: 承接 2026-06-26 主题令牌体系重构（commit `d441ef2`）

## 1. 背景与目标

### 1.1 问题

经对前端 100+ 个样式文件的量化扫描，确认视觉"不统一、不对齐"的三类技术根因：

- **面板 header 高度混乱**：硬编码值散布于 53/56/57/44/48/40/36px 等十余种，导致相邻面板顶部无法对齐。
- **圆角值失控**：`border-radius` 硬编码 9 种并存（2/3/4/6/8/10/12/14/16px），同一应用视觉风格不一致。
- **字号阶梯膨胀**：`font-size` 硬编码 15 种（8~36px），视觉层级模糊。

**核心症结**：令牌体系（376 个：primitive 124 + semantic 125 + component 127）已成熟，但组件层只接入了一一半——172 处硬编码间距 vs 158 处令牌引用（≈1:1），导致每个面板"各自为政"。

### 1.2 目标

借机全面修缮前端视觉，对必要处重构、对瑕疵处修缮：

- **统一**面板结构（header/body/footer）与控件规范，消除"不对齐"
- **收敛**散乱的圆角/字号/间距碎值到标准阶梯，消除"不统一"
- **最小占位**：拒绝纯机械替换（如 `8px → var(--ui-space-sm)` 而值本就相等），只在有视觉收益处令牌化

### 1.3 非目标

- **不做视觉风格升级（C 阶段）**：间距阶梯重定义、阴影层级、密度调整等设计语言层面的改动留待下一阶段，避免一步到位的返工。
- **不改画布节点（`nodes/` 子系统）**：Vue Flow 节点结构特殊（handle/edge），属独立子系统，本次不碰。
- **不改 `inspectors/configDriven/renderers/`**：渲染器数量多且趋同，留待后续批量处理。
- **不重设计整体视觉语言**：配色/字体/组件库重写超出"修缮"范畴。

### 1.4 产品定位

**非专业性质的数据平台**，视觉基调偏**友好温和**（消费产品感），而非 SaaS 工具的紧凑专业感。此定位驱动了后续所有数值决策。

## 2. 需求决策记录

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 重构范围 | **B：令牌收敛 + 面板布局统一** | 精准对症"不统一/不对齐"，C 阶段留后 |
| 视觉基调 | **友好温和**（非专业数据平台） | 贯穿所有数值决策 |
| Header 密度 | **宽松型 56px** | 呼吸感强，契合友好基调 |
| 圆角阶梯 | **圆润型 8/12/16/24** | 偏消费产品友好感 |
| 间距基准 | **8px 基准**（4/8/16/24/32/48） | 呼吸感强，跳过 12px 减少碎值 |
| 执行策略 | **B 为主（面板闭环）+ C 为辅（面板内 header→圆角→间距）** | 可验证可回滚 |
| 令牌策略 | **方案 2 收敛版**：新增 1 个令牌 + radius 改值 + 组件层接入 | 最小占位，拒绝纯机械替换 |
| Radius 令牌值 | **改值到 8/12/16/24**（接受全局连锁） | 强制统一，分主题验证控险 |
| Header 字号 | **统一 14px**（`--ui-font-size-md`），不新增专用令牌 | 强制统一，先不考虑特例 |

## 3. 令牌规范（第 1 节）

### 3.1 新增令牌（仅 1 个）

| 令牌 | 值 | 定义位置 | 用途 |
|------|-----|---------|------|
| `--ui-header-height` | 56px | `tokens/component.css` | 所有面板 header 统一高度基准 |

> 控件高度经评估可复用现有体系（基类已 36px），不强加令牌。现有 `--space-*`、`--font-size-*` 阶梯全部复用，不新增。

### 3.2 Radius 规范（改令牌值，全局收敛）

现有令牌值 → 新值：

| 令牌 | 现值 | 新值 | 用途 |
|------|------|------|------|
| `--radius-sm` | 8px | **8px**（不变） | 控件：按钮、输入框、标签、badge |
| `--radius-md` | 10px | **12px** | 小卡片、下拉菜单、弹层 |
| `--radius-lg` | 14px | **16px** | 面板容器、大卡片 |
| `--radius-xl` | 20px | **24px** | 模态框、大型容器 |
| `--radius-full` | 9999px | 9999px（不变） | 圆形元素、pill 标签 |

**影响面**：358 处 `var(--ui-radius-*)` 引用 + 22 处 `var(--radius-*)` 引用一次性收敛。改值后必须在 light/dark/liquid 三主题下逐一视觉验证。

### 3.3 间距规范（8px 基准，复用现有阶梯，不改值）

| 语义令牌 | 值 | 用途 |
|---------|-----|------|
| `--ui-space-xs` | 4px | 紧凑间距：图标-文字、badge 内距 |
| `--ui-space-sm` | 8px | 控件内部：input 内距、按钮内距 |
| `--ui-space-md` | 12px | 默认间距：表单项间、列表项间 |
| `--ui-space-lg` | 16px | 区块内距：panel body、卡片内距 |
| `--ui-space-xl` | 20px | 区块间距：header padding、大段落间距 |
| `--ui-space-2xl` | 24px | 大分隔：section 间距 |

### 3.4 碎值映射规范（消除散乱，不增令牌）

组件层把偏离阶梯的硬编码碎值映射到最近标准值。**裁决原则：友好基调偏宽松，遇歧义（如 10px）一律向上取整到 12px**，不降级到 8px：

```
现状硬编码碎值   →   映射目标
1, 2, 3 px       →   4px  (--ui-space-xs)
5, 6, 7 px       →   8px  (--ui-space-sm)    [偏宽松，不降为4]
9, 10 px         →   12px (--ui-space-md)    [向上取整，不降为8]
14, 18 px        →   16px (--ui-space-lg)
```

### 3.5 字号规范（复用现有阶梯，收敛散乱）

| 语义 | 令牌 | 值 | 用途 |
|------|------|-----|------|
| 极小 | `--ui-font-size-xs` | 10px | 时间戳、辅助标签 |
| 小 | `--ui-font-size-sm` | 12px | 次要文字、meta、表格副文本 |
| 正文 | `--ui-font-size-md` | 14px | 正文、控件文字、表格主文本、**panel header 标题** |
| 大 | `--ui-font-size-lg` | 16px | 区块标题、强调 |
| 更大 | `--ui-font-size-xl` | 18px | 页面标题 |

**收敛规则**（同样遵循"向上取整、偏宽松"原则）：当前散乱的 8/9px → 10px；11px → 12px；13px → 14px；**消除 15px**（并入 14px）。

## 4. 面板结构统一规范（第 2 节）

所有面板遵循统一结构模式：**Header + Body + 可选 Footer**。

| 区域 | 规范 | 令牌引用 |
|------|------|---------|
| **Header** | 56px 高 · padding 0 20px · 标题 14px/font-weight 600 · 图标 18×18px · 底部 1px 分隔线 · 背景 elevated | `height: var(--ui-header-height)`<br>`padding: 0 var(--ui-space-xl)`<br>`font-size: var(--ui-font-size-md)` |
| **Body** | padding 16px · 子项间距 12px · flex:1 + overflow-y:auto | `padding: var(--ui-space-lg)`<br>`gap: var(--ui-space-md)` |
| **Footer**（仅操作类面板：Inspector、Modal） | 48px 高 · 顶部分隔线 · padding 0 16px · gap 8px | `height: 48px`<br>`padding: 0 var(--ui-space-lg)` |

**对齐保证**：header 统一 56px 后，相邻面板顶部严格对齐到同一水平线——这是"对齐感"的核心。

## 5. 控件统一规范（第 3 节）

### 5.1 高度阶梯（3 档）

| 档位 | 高度 | 用途 | 字号 |
|------|------|------|------|
| **sm** | 32px | 密集工具栏、表格内操作 | 12px (`--ui-font-size-sm`) |
| **md** ✓默认 | 36px | 主操作、表单按钮 | 14px (`--ui-font-size-md`) |
| **lg** | 40px | 模态框主按钮、空状态 CTA | 14px (`--ui-font-size-md`) |

### 5.2 映射规则

```
28, 30, 32 px →  32px (sm)
34 px         →  36px (md)
36 px         →  36px (md)  [基准，ui.css 已符合]
38, 40 px     →  40px (lg)
```

### 5.3 配套规范

- **圆角**：统一 `--ui-radius-sm`（8px）
- **按钮内距**：`padding 0 var(--ui-space-md)`（12px）；sm 档用 `var(--ui-space-sm)`（8px）
- **输入框内距**：`padding 0 var(--ui-space-md)`（12px）
- **图标-文字 gap**：`var(--ui-space-sm)`（8px）

## 6. 面板优先级与范围清单（第 4 节）

### 6.1 分批规划

| 批次 | 范围 | 硬编码密度 | 理由 |
|------|------|-----------|------|
| **P0 基础** | `tokens/` 令牌层 + `ui.css` 控件基类 + `app-shell.css` 壳体 | — | 地基：令牌值调整 + 基类规范，后续面板依赖 |
| **P1 核心面板** | InspectorPanel · DataLibrary · ResourceTree · ToolboxPanel | 15/39/28/12 魔数 | 主工作区高频可见 |
| **P2 高频弹层** | SettingsModal · ProjectManagementModal · AssetLibraryNav | 24/7/20 魔数 | 用户频繁打开 |
| **P3 辅助面板** | AIChatPanel · ResourceExplorerPanel · ToolboxTile · 其余 inspector 子组件 | 53/4/5/... | 使用频率稍低或单点问题 |
| **不本次改** | `nodes/` 画布节点 · `inspectors/configDriven/renderers/` | 量大 | 独立子系统，留待评估 |

### 6.2 P0 关键改动

1. **Radius 令牌值调整**：`--radius-sm/md/lg/xl` 8→8 / 10→12 / 14→16 / 20→24（在 `tokens/primitive.css`）
2. **新增** `--ui-header-height: 56px`（`tokens/component.css`）
3. **ui.css 控件基类**：确认 `.ui-btn`/`.ui-input` 等已对齐 md 36px / radius-sm 8px（查漏补缺）
4. **app-shell.css**：activity-bar/sidebar/panel 三栏顶部对齐保证

### 6.3 验证策略

- **每批完成后**：在 light/dark/liquid 三主题下视觉验证
- **P0 完成是关键检查点**：radius 改值触发全应用连锁，此时做第一次完整验证；不通过则回滚令牌值
- **回归手段**：E2E 测试（roundtrip/validation）验证功能不受影响；视觉问题靠人工 + 截图对比

## 7. 执行约定

### 7.1 处理级别（拒绝无效占位）

| 级别 | 做什么 | 例子 |
|------|--------|------|
| **必改**（有视觉统一收益） | header 高度对齐 `--ui-header-height`；header padding/font 统一 | `53px → var(--ui-header-height)` |
| **应改**（消除碎值） | 偏离阶梯的碎值映射到最近标准值 | `10px → 8px 或 12px` |
| **不改**（避免无效占位） | 值已等于令牌值且无统一诉求的硬编码，不强行套令牌 | `padding: 8px` 视觉即 8px 且无需统一时保留 |

### 7.2 面板内处理顺序（C 为辅）

每个面板闭环内，按固定顺序处理：**header → 圆角 → 间距 → 字号**，避免遗漏。

### 7.3 兼容性

- Radius 改值会触发 358+22 处引用连锁——这是预期的全局收敛，非回归
- 令牌层改动需同步检查 `tokens/compat.css` 中 `--ui-radius-*` 映射（指向 `--radius-*`，会自动跟随）
- 主题文件 `themes/dark.css` / `themes/liquid.css` 中若有 radius 覆盖需一并检查

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Radius 改值导致全局视觉波动 | P0 后三主题完整验证；不通过则回滚令牌值（单点回退） |
| 碎值映射上下文判断失误（如 10px 该升 12 还是降 8） | 遵循"友好基调偏宽松"原则，疑难点在验证阶段人工裁决 |
| 面板数量多导致遗漏 | 按 P0→P3 批次推进，每批独立验证 |
| 改动量大引入功能回归 | E2E 测试守护；纯 CSS 改动理论上不影响功能逻辑 |
